# Дорожная карта реализации

Telegram-бот приёма заказов «ЛСП Город STL файлы» (заказчик — Zurab). Клиент шлёт
файлы боту → бот собирает их в один заказ → карточка + файлы уходят в закрытую
группу «Город», где владелец, админ и оператор. Оператор скачивает файлы вручную.

6 этапов от пустого репозитория до прод-готовности. Каждый этап заканчивается
**демонстрируемым результатом** (что можно потрогать руками).

Ключевая идея: **MVP-first**. Этап 1 = живая вертикаль сбора заказа в боте.
Всё остальное наслаивается без перепиливания базы.

**Стек** (как в `sub_bot`): Python + aiogram 3, Postgres 16 (отдельная схема, локальную
БД не трогаем), Redis (FSM-сценарий), Docker, polling, автомиграции, полный слой
логирования в терминал, CI/CD через GitHub Actions.

**Ключевые ограничения из ТЗ / переписки**:
- Максимальная простота. Никаких статусов, кнопки «взять в работу», админки, поиска в боте.
- Перед приёмом файлов — предупреждение: файлы после 19:00 могут быть изготовлены через
  двое суток + отказ от ответственности за повреждённые/некорректные файлы.
- Заказ создаётся в любом случае (заказчики невнимательные); техник может дослать/исправить файлы.
- Утечка информации: внутренние сотрудники не должны видеть внешние заказы —
  поэтому отдельная закрытая группа «Город», а не общая.
- Большие файлы (редко, видео до 120 МБ): сначала пробуем метод пересылки для обхода
  лимита 50 МБ, fallback — заливка в S3 (Beget) и ссылка в карточке.

> **Где смотреть прогресс**: текущий статус всех этапов — в [`STATUS.md`](STATUS.md).
> Внутри каждого этапа ниже — чекбоксы конкретных acceptance-пунктов. Закрытие этапа
> сопровождается git-тегом `stage-N-done`.

| # | Этап | Главный артефакт | Зависит от |
|---|---|---|---|
| 0 | Каркас репозитория | Бот запускается (polling), `/start` отвечает, БД-схема + Redis + миграции подняты | — |
| 1 | Сбор заказа в боте | Полный FSM: Новый заказ → предупреждение → файлы → Завершить → имя → кол-во → «Принят», заказ в БД | 0 |
| 2 | Карточка в группу «Город» | Карточка (№, имя, кол-во) + все файлы реально прилетают в закрытую группу | 1 |
| 3 | Большие файлы (>50 МБ) | Файл >50 МБ доходит до группы (пересылка или S3-ссылка) | 2 |
| 4 | Docker + CI/CD + деплой | Бот крутится на сервере 45.88.14.4, автодеплой по пушу в GitHub | 3 |
| 5 | Инструкция и приёмка | Краткая инструкция по использованию, финальная сквозная проверка | 4 |

## Этап 0 — Каркас репозитория

**Цель**: пустой, но рабочий проект на базе `sub_bot`, очищенный от логики
лид-магнита/подписок, в котором есть всё для следующих этапов.

**Что делаем**:
- Копируем скелет `sub_bot` в проект, вычищаем лид-магнит: подписки, подарок, напоминалки,
  проверку канала (`services/subscription.py`, `services/gift.py`, `services/channel.py`,
  `scheduler.py` и связанные хендлеры/тексты).
- Свой git-репозиторий (публичный на GitHub — `dimatolpygin/duntist_bot`).
- `config.py` под новые переменные: `BOT_TOKEN`, `GROUP_ID` (группа «Город»), `DB_SCHEMA`,
  S3-доступы; `.env.example` обновлён.
- Postgres: отдельная схема (напр. `zarub`), локальную БД и чужие схемы не трогаем.
- Redis для FSM.
- Миграция `0001_init.sql`: таблица заказов (`orders`) + при необходимости `order_files`.
- Автоприменение миграций при старте.
- Полный слой логирования в терминал (каждое действие пользователя: username, id, first_name, текст).
- Dockerfile + docker-compose (dev) поднимают бота.

**Критерии приёмки** (закрытие → tag `stage-0-done`):
- [ ] `docker compose -f docker-compose.dev.yml up` поднимает бота без ошибок
- [ ] Бот подключается к Postgres, создаёт свою схему и применяет миграции (в логе видно)
- [ ] Бот подключается к Redis
- [ ] `/start` в боте отвечает приветствием с кнопкой «Новый заказ» (без логики лид-магнита)
- [ ] В логе терминала видно действие пользователя (username, id, first_name, текст)
- [ ] Код лид-магнита (подписки/подарок/напоминалки) полностью удалён из проекта

## Этап 1 — Сбор заказа в боте

**Цель**: техник может целиком собрать заказ внутри бота (без отправки в группу).

**Что делаем**:
- Кнопка «Новый заказ» → показ предупреждения (сроки после 19:00 + ответственность за
  повреждённые файлы) перед приёмом файлов.
- FSM-сценарий приёма: клиент шлёт любое количество файлов/фото/архивов/документов;
  бот их накапливает в рамках одного заказа (ограничений по количеству нет).
- Кнопка «Завершить заказ» → бот спрашивает имя клиента → количество/описание.
- Бот подтверждает: «Заказ успешно принят», присваивает номер.
- Заказ и связанные файлы сохраняются в БД (schema `zarub`): номер, имя, кол-во/описание,
  список file_id, автор (tg id/username).
- Обработка «забыл нажать Завершить»: заказ всё равно можно завершить; файлы не теряются.

**Критерии приёмки** (закрытие → tag `stage-1-done`):
- [ ] «Новый заказ» показывает предупреждение про 19:00 и ответственность до приёма файлов
- [ ] Клиент отправляет несколько файлов разных типов — все принимаются в один заказ
- [ ] «Завершить заказ» → бот спрашивает имя, затем количество/описание
- [ ] После ввода бот отвечает «Заказ успешно принят» с номером
- [ ] В таблице `zarub.orders` появляется запись с номером, именем, кол-вом и списком файлов
- [ ] Второй заказ получает следующий по порядку номер

## Этап 2 — Карточка в группу «Город»

**Цель**: готовый заказ уходит одной карточкой + файлами в закрытую группу.

**Что делаем**:
- После «Заказ принят» бот шлёт в группу `GROUP_ID` карточку: № заказа, имя клиента,
  количество/описание (максимально просто).
- Следом отправляет все файлы заказа, привязанные к этому заказу.
- Дослать/исправить файлы после завершения — файлы долетают в ту же группу с пометкой к заказу.

**Критерии приёмки** (закрытие → tag `stage-2-done`):
- [ ] В группе «Город» появляется карточка с №, именем клиента и количеством/описанием
- [ ] Следом в группе появляются все файлы этого заказа
- [ ] Клиент получает подтверждение, что заказ принят
- [ ] Внешний клиент не видит содержимое группы (шлёт только боту)

## Этап 3 — Большие файлы (>50 МБ)

**Цель**: файлы больше лимита Telegram доходят до группы.

**Что делаем**:
- Определяем размер файла; для файлов в пределах лимита — обычная пересылка/копия.
- Для больших: сначала пробуем метод пересылки (copy/forward file_id, полученного ботом,
  обходит перезаливку 50 МБ), fallback — заливка в S3 (Beget) и ссылка в карточке.
- Тест на видео ~120 МБ (ссылку на S3 заказчик пришлёт).

**Критерии приёмки** (закрытие → tag `stage-3-done`):
- [ ] Файл в пределах лимита доходит до группы как файл
- [ ] Файл >50 МБ доходит до группы: либо переслан, либо ссылкой на S3
- [ ] В карточке/сообщении понятно, где взять большой файл
- [ ] Ошибка заливки не роняет бот — заказ всё равно создаётся, ошибка логируется

## Этап 4 — Docker + CI/CD + деплой

**Цель**: бот работает на сервере, обновляется автоматически по пушу.

**Что делаем**:
- Prod `docker-compose.yml` (бот + при необходимости Postgres/Redis, либо внешние).
- GitHub Actions: сборка/деплой по пушу в целевую ветку.
- `scripts/install.sh` автоустановки на чистой Ubuntu (по образцу РЕСТОРАН/sub_bot).
- Деплой на сервер `root@45.88.14.4`.
- Список Secrets для GitHub-репо и гайд по deploy-ключу.

**Критерии приёмки** (закрытие → tag `stage-4-done`):
- [ ] Бот запущен на сервере 45.88.14.4 и отвечает в Telegram
- [ ] Пуш в GitHub-репо триггерит автодеплой (Actions зелёный, бот перезапустился)
- [ ] `install.sh` разворачивает проект с нуля по ссылке на репо
- [ ] Гайд по Secrets и deploy-ключу готов

## Этап 5 — Инструкция и приёмка

**Цель**: заказчик может пользоваться ботом, финальная сквозная проверка пройдена.

**Что делаем**:
- Краткая инструкция по использованию (для клиента и для оператора/админа).
- Сквозной прогон: клиент → заказ → группа → скачивание оператором.

**Критерии приёмки** (закрытие → tag `stage-5-done`):
- [ ] Краткая инструкция по использованию написана
- [ ] Сквозной сценарий пройден вживую и подтверждён заказчиком
- [ ] Известные ограничения зафиксированы (что не входит в базовую версию)
