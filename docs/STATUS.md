## История закрытий

Записывать в обратном хронологическом порядке. Формат:
`YYYY-MM-DD — Этап N: <название> — tag stage-N-done — короткая заметка с конкретикой верификации`.

- 2026-07-06 — Этап 3: Большие файлы (>50 МБ) — tag `stage-3-done` — основной механизм — пересылка по `file_id` (обходит лимит upload 50 МБ, тянет 120 МБ), подтверждено заказчиком («ок с большими файлами»). S3-fallback (`services/s3.py`, boto3) для сбойных отправок ≤20 МБ (лимит getFile) + заметка оператору для больших. S3-путь провалидирован: заливка на Beget → публичная ссылка HTTP 200 → удаление объекта. Файлы >20 МБ, не прошедшие пересылку, S3-fallback не покрывает (нужен локальный Bot API, вне базовой версии) — на практике пересылка их и так доставляет.
- 2026-07-06 — Этап 2: Карточка в группу «Город» — tag `stage-2-done` — сервис `services/delivery.py`: после «Заказ принят» бот шлёт в `GROUP_ID` карточку (№, клиент, количество, файлов, отправитель) + все файлы заказа по `file_id` (пересылка без перезагрузки, обходит лимит upload 50 МБ). Ошибки доставки изолированы от клиентского потока. UAT: заказ №5 доставлен `карточка + 1/1 файлов`, ошибок нет. Правка по обратной связи: строка «Отправитель» теперь ник + first_name + user_id.
- 2026-07-06 — Этап 1: Сбор заказа в боте — tag `stage-1-done` — FSM-сценарий (`handlers/order.py`): «Новый заказ» → предупреждение (без кнопок) → приём файлов → «Завершить» → имя → количество → «Заказ №N принят», запись в `zarub.orders`/`order_files`. На UAT всплыли и починены баги пачечной загрузки: (1) гонка при альбоме теряла файлы (счёт 1,2,3,3,3) — добавлен per-user `asyncio.Lock` на добавление файла; (2) спам подтверждений на каждый файл — дебаунс 0.8с в одно сообщение; (3) два подтверждения от разных альбомов (docx и фото шлются раздельно) с дублем кнопок — теперь держится одно сообщение-счётчик (предыдущее удаляется), кнопка только на нём. Проверено в БД: заказ №1 = 5 файлов (doc+3 photo+video), заказ №2 = 8 файлов (5 doc+3 photo), номера по порядку, имя/количество сохранены.

## Протокол обновления

**При старте сессии**:
1. Прочитать этот файл.
2. `git tag -l "stage-*"` и сверить с колонкой «Закрывающий тег». Тег без записи в таблице = таблица устарела, обновить.
3. `git log --oneline -10` — что менялось последним.
4. Только после этого приступать к работе.

**При взятии этапа в работу**:
1. Обновить «Текущий этап» и «Активная работа».
2. Статус 🚧 в таблице.
3. Закоммитить отдельно: `chore(status): start stage N`.

**При закрытии этапа**:
1. Убедиться, что все чекбоксы в `07_ROADMAP.md` отмечены `[x]` и реально проверены человеком.
2. Создать тег: `git tag stage-N-done -m "<описание>"`.
3. В таблице: ✅, «Закрывающий тег», «Последний коммит» (`git rev-parse --short HEAD`), «Дата закрытия».
4. Добавить запись в «Историю закрытий».
5. Указать новый «Текущий этап» и «Следующий шаг».
6. Закоммитить: `chore(status): close stage N`.

**Если этап заблокирован**:
1. Статус ⏸ в таблице.
2. В «Известные блокеры»: что блокирует, кого ждём, что нужно для разблокировки.
